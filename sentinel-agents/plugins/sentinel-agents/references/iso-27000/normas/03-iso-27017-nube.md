# ISO/IEC 27017 — Requerimientos para la Nube

> **Checklist asociado:** [`../checklists/checklist-27017.md`](../checklists/checklist-27017.md)
> **Enfoque:** Extiende la 27002 con controles específicos para arquitecturas virtuales e híbridas.

---

## 1. Qué es y por qué existe

La 27002 asume infraestructura tradicional. La **27017 añade los controles que solo tienen sentido en la nube**: virtualización, multitenancy, consolas de administración y APIs de proveedor. También define el **modelo de responsabilidad compartida**: qué asegura el proveedor de nube (CSP) y qué asegura el cliente.

```
   RESPONSABILIDAD DEL CLIENTE          RESPONSABILIDAD DEL PROVEEDOR
   ──────────────────────────          ─────────────────────────────
   Datos, IAM, configuración,          Hipervisor, hardware físico,
   apps, parches del SO (IaaS),        red física, disponibilidad
   cifrado de sus datos                del datacenter
```

---

## 2. Aislamiento de Entornos (*Multitenancy*)

**Objetivo:** que los recursos de un cliente no se mezclen con los de otro.

Si eres **proveedor**, debes demostrar que los datos y recursos de cómputo (CPU/RAM) del **Cliente A están lógicamente separados** de los del Cliente B, a nivel de:
- **Hipervisor** (aislamiento de VMs),
- **Kubernetes** (namespaces, *network policies*, *resource quotas*, *pod security*).

Si eres **cliente**, debes verificar que el proveedor documenta y certifica ese aislamiento.

---

## 3. Hardening de Contenedores e Imágenes

**Objetivo:** desplegar solo imágenes seguras y reproducibles.

- Configuración segura de instancias mediante plantillas **AMIs / Golden Images** revisadas y versionadas.
- **Escaneo de vulnerabilidades en imágenes Docker antes de desplegarlas** (ej. Trivy, Grype, Clair).
- Imágenes mínimas (*distroless* / *slim*), sin paquetes ni shells innecesarios.
- No ejecutar contenedores como **root**; usar usuarios sin privilegios.

```
Build imagen ──► Escaneo (Trivy) ──► ¿Vulns críticas? ──Sí──► BLOQUEAR
                                          │No
                                          ▼
                                    Push al registro ──► Deploy
```

---

## 4. Seguridad en la Consola de Administración

**Objetivo:** proteger el panel que controla toda la infraestructura.

- Restringir el acceso a la consola de **AWS/Azure/GCP por IPs específicas** (whitelisting).
- Logs de auditoría de infraestructura **siempre activos** (ej. **AWS CloudTrail**, Azure Activity Log, GCP Cloud Audit Logs).
- **Eliminar el uso de cuentas raíz (root)**: bloquearla, ponerle MFA y operar con roles de mínimo privilegio.
- MFA obligatorio para todo acceso administrativo.

---

## 5. Sincronización de Relojes (NTP)

**Objetivo:** que los logs de todos los sistemas coincidan en el tiempo.

Configuración **obligatoria de NTP** en todas las instancias virtuales. Si los relojes están desfasados, la **línea de tiempo de una investigación forense es inservible** (no puedes correlacionar eventos entre sistemas).

```
   Sin NTP:  Servidor A 10:00:03   Servidor B 09:58:41   ← imposible correlacionar
   Con NTP:  Servidor A 10:00:03   Servidor B 10:00:03   ← eventos alineables al ms
```

---

## 6. Resumen de controles clave

| Control | Requisito verificable |
|---------|------------------------|
| Multitenancy | Aislamiento documentado a nivel hipervisor/K8s |
| Imágenes | Escaneo obligatorio pre-deploy + no-root |
| Consola admin | IP whitelisting + CloudTrail + sin root |
| NTP | Sincronización activa en todas las instancias |
